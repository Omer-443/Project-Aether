A. AUDIT OF `frontend/app/visualizer/page.tsx`
==============================================

A1) High-level purpose (what this page is “for”)
--------------------------------------------------
The Visualizer page is a client-side interactive UI that:
- Fetches a continuous trajectory telemetry dataset from the backend.
- Renders it as a Recharts line chart (“Oscilloscope Curve”).
- Shows additional derived “gap markers” (ReferenceLines).
- Computes client-side stats such as:
  - current (latest z)
  - peak (max z)
  - volatility (a standard-deviation-like metric over z)
  - events (derived from backend stats or fallback)
- Provides UI controls to:
  - Play/Pause chart line animation
  - Reset chart settings
  - Modify “intensity”, which changes how many trajectory events/steps are requested (nEvents)
  - Manually inject more “window” (extend/widen/reset), again changing nEvents.

So: its actual role is “trajectory visualization and derived stats,” not data ingestion and not end-to-end policy simulation.


A2) Client/runtime characteristics
-----------------------------------
- Declared at top: 'use client' → runs in the browser.
- Uses React hooks: useEffect, useMemo, useState.
- Uses Framer Motion for animations.
- Uses Recharts for charting.
- Has lots of debug logging (console.log / console.error) inside the data fetching effect.


A3) Data model types expected by this page
-------------------------------------------
The file includes TypeScript types for:
- TrajectoryTelemetryResponse (the API payload the backend returns)
- RiskLabel union: 'LOW' | 'MODERATE' | 'HIGH' | 'CRITICAL'
- GapMarker: { time:number, label:string, days:number }

Critical expected fields:
- data.provider (string)
- data.risk_label (enum-like string)
- data.trajectory[] items containing at least:
  - time:number
  - z:number
  - friction:number
  - plus additional dim_0..dim_3 (the frontend does not use them directly, but backend may include them)
- data.gaps[] items containing:
  - time:number
- data.stats.event_count (frontend tries to use it)


A4) State variables and what they control
------------------------------------------
- isPlaying: boolean
  - Controls Recharts Line animation behavior (isAnimationActive, animationDuration)

- intensity: number (0..100 slider)
  - Not used directly in chart data.
  - Instead it changes nEvents via requestFromIntensity.

- nEvents: number
  - Controls the API request parameter: n_events
  - Also used to display event count fallback.

- state: { data, loading, error }
  - data is TrajectoryTelemetryResponse | null


A5) Effects & when API calls happen
------------------------------------
Key effect:
- useEffect depends on [nEvents]
- On mount and whenever nEvents changes:
  1) setState loading=true
  2) call getTrajectoryTelemetry(nEvents)
  3) setState with received telemetry
  4) catch sets error state

Cancellation:
- A `cancelled` boolean prevents setState if the component unmounts (or effect re-runs before completion).

Debug logging:
- Logs include:
  - nEvents values
  - telemetry received lengths
  - provider/risk_label
  - first and last trajectory sample

Net behavior:
- Any nEvents change triggers a backend call.


A6) Intensity slider mapping → nEvents
--------------------------------------
- requestFromIntensity(value)
  - sets intensity
  - sets nEvents = clampEventCount(4 + (value / 100) * 24)

clampEventCount(value):
- Math.max(2, Math.min(36, Math.round(value)))

So intensity effectively maps to:
- a computed range roughly 4..28 before clamp
- but clamped to 2..36 after rounding

Important: there is no debounce/throttle.
- Because onChange triggers while dragging, this can spam backend requests.


A7) Chart data transformation (chartData)
------------------------------------------
chartData is derived from state.data.trajectory.
For each point from backend trajectory:
- x: Number(point.time).toFixed(2)
  - NOTE: toFixed returns a STRING.
- z: Math.round(Number(point.z ?? 0) * 10000) / 100
  - => effectively two-decimal rounding
- friction: similar rounding from point.friction

So the chartData format becomes:
- { x: string, z: number, friction: number }

The page uses chartData to render:
- <XAxis dataKey="x" ... />
- <Line dataKey="z" ... />
- <Line dataKey="friction" ... />


A8) Gap markers mapping (gapLines)
-----------------------------------
gapLines derived from state.data.gaps:
- (state.data.gaps ?? []).map(gap => Number(gap.time).toFixed(2))

Again:
- toFixed returns string
- so ReferenceLine x is passed string values.


A9) Stats computation (stats)
--------------------------------
stats useMemo:
- zValues = chartData.map(p => p.z).filter(Number.isFinite)
- current = last z value or null
- peak = max z value or null
- volatility = volatility(zValues)

volatility(values):
- If values length < 2 → 0
- mean
- variance = average squared deviation (divide by values.length)
- sqrt(variance)
- returned as a number, later displayed with “%” suffix:
  - <div> {stats.volatility.toFixed(2)}% </div>

Events metric:
- events: state.data?.stats?.event_count
        ?? state.data?.events?.length
        ?? nEvents

This depends on backend providing stats.event_count.


A10) UI controls and their functional impact
---------------------------------------------
1) Play/Pause button
- toggles isPlaying
- only affects Recharts animation properties
- it does NOT change data or time progression.

2) Reset button
- sets isPlaying=false
- sets intensity=50
- sets nEvents=12

This triggers effect due to nEvents update.

3) “Event Injection” buttons
- Extend Window: setNEvents(value => clampEventCount(value + 1))
- Widen Gap: setNEvents(value => clampEventCount(value + 4))
- Reset Window: setNEvents(12)

These update nEvents → triggers backend calls.


A11) Backend endpoint actually used by Visualizer
--------------------------------------------------
Frontend method call:
- getTrajectoryTelemetry(nEvents)
from frontend/lib/api/aether-client.ts.

That method does:
- apiGet('/api/v1/telemetry/trajectory', { n_events: nEvents })
- includes credentials: 'include'
- parses JSON response
- throws detailed errors if res.ok is false

Therefore Visualizer actively serves content only if:
- backend is reachable at DEFAULT_BASE_URL (http://localhost:8000)
  OR NEXT_PUBLIC_AETHER_API_BASE_URL is set.
- CORS/credentials policy allows requests.


A12) Backend implementation of `/api/v1/telemetry/trajectory`
---------------------------------------------------------------
In backend/app/api/routes.py:
- route: @router.get("/telemetry/trajectory", response_model=TrajectoryTelemetryResponse)
- query param: n_events: int = 12

Backend behavior:
1) sample = _sample_provider_events(n_events, seed=99)
2) events = sample["events"]
3) model = get_model()
4) coeffs, times, _ = coeffs_from_claim_events(events)
5) with torch.no_grad(): result = model(coeffs, return_trajectory=True)
6) denial_prob = round(result['denial_prob'].item(), 4)
7) raw_trajectory = _trajectory_to_json(result['trajectory'].tolist(), result['eval_times'].tolist(), n_dims=4)
   - _trajectory_to_json exports points as:
     { time: round(t,2), dim_0..dim_3 } but with only first n_dims.
8) return TrajectoryTelemetryResponse(...)
   - trajectory = _frontend_trajectory(raw_trajectory, denial_prob)
     This converts latent dims into:
     - time: float(point['time'] - t0) rounded to 2 decimals
     - z: sigmoid(latent dim_0)
     - friction: bounded function of denial_prob and signal
     - dim_0..dim_3
   - gaps = _gap_markers_from_events(events)
     - computes date diffs and creates GapMarkers when diff >= 21 days.
   - risk_label = _risk_label(denial_prob)
     - thresholds: <0.25 LOW, <0.55 MODERATE, <0.75 HIGH else CRITICAL
   - stats contains:
     - time_steps
     - latent_dims
     - irregular_gaps count
     - event_count
     - time_start/time_end


A13) Is Visualizer “needed” / does it provide real value?
--------------------------------------------------------
What it provides:
- A continuous curve view and “friction” overlay
- Visualization of risk label context

What it is NOT:
- Not a simulator UI (it only visualizes trajectory telemetry for a sampled event set)
- Not necessarily user-input-driven.

In current backend design:
- telemetry is derived from deterministic sampling with seed=99.
- That means visualizer is more “demo/prototype exploration” than “user-run telemetry.”


A14) Behavioral issues / shortcomings (performance & robustness)
--------------------------------------------------------------
1) API call frequency during slider drag
- intensity slider triggers onChange continuously.
- Each change updates nEvents, effect runs, new fetch starts.
- No debounce → high backend load risk.

2) Chart x-axis uses strings
- x = toFixed(2) returns string.
- ReferenceLine x uses also string.
- Recharts may work, but numeric domain/ordering can behave less predictably.

3) “Play” is only animation, not simulation time
- Data is static after fetch.

4) LineChart key forces remount
- key includes nEvents and trajectory length.
- Remounting can lose internal chart animation state.

5) Volatility displayed with “%” suffix
- But volatility() is sqrt(variance) of z.
- Unless z is percent-like already, “%” labeling may be misleading.



B. AUDIT OF `frontend/app/logs/page.tsx` (“purpose of last log/page.tsx”)
======================================================================

B1) High-level purpose
-----------------------
This page renders a UI table of “Raw Telemetry Logs” with:
- client-side search
- client-side status filtering
- an automatic refresh every 30 seconds
- a few summary counters (Total, Denied, Pending, Avg Friction)

It is essentially a diagnostic/monitoring view for telemetry rows.


B2) How it fetches data
------------------------
- Declared 'use client'
- On mount (useEffect with [] dependency):
  - calls getLogsTelemetry(80)
  - maps response rows into a simplified display model
  - sets interval to reload every 30 seconds

Fetch cancellation:
- uses cancelled boolean similar to visualizer.


B3) What it expects from backend
---------------------------------
- It imports getLogsTelemetry and type LogsRow from frontend/lib/api/aether-client.ts.

Type mapping used in this page:
- DisplayLog includes:
  - time: string (uses r.timestamp)
  - level: LogsRow['status'] (r.status)
  - module: string (hardcoded as `CLAIM_${r.status}`)
  - action: string (r.cpt_opaque_id)
  - friction: number (Number(r.friction_score))

So backend must return:
- rows[] items with:
  - id
  - timestamp
  - cpt_opaque_id
  - friction_score
  - status


B4) Backend endpoint powering logs
-----------------------------------
frontend/lib/api/aether-client.ts provides:
- getLogsTelemetry(limit)
  - GET '/api/v1/telemetry/logs' with query { limit }

In backend/app/api/routes.py:
- @router.get("/telemetry/logs", response_model=LogsTelemetryResponse)
- signature: limit: int = 80

Backend behavior summary:
- loads CMS data: load_cms_data(n_rows=max(limit,80), seed=321)
- sorts by claim_date descending
- selects head(limit)
- constructs synthetic “log rows” by computing:
  - continuous_time in days since earliest claim_date
  - friction score from prior_denial_rate and payer_score
  - latent_state from friction and drg_weight
  - status assigned pseudo-deterministically:
    - if denied==1 → status DENIED
    - else status based on procedure_count and index cycling
- returns rows and summary counts

Important: this is computed telemetry derived from loaded mock dataset, not raw event streaming.


B5) What the logs page is NOT
-------------------------------
- It is not “live system logs” from a queue.
- It is not “user-run correlated logs.”
- It does not accept a run_id or filter by run.
- It does not use backend-provided summary field.

D. “WHAT’S THE PURPOSE” CONCLUSION (neutral)
===============================================

- Visualizer page purpose: show continuous trajectory + derived friction + gap markers; fetches telemetry via `/api/v1/telemetry/trajectory`.
- Logs page purpose: show a dense table of telemetry rows with client-side search/filter; refreshes every 30 seconds via `/api/v1/telemetry/logs`.

Both are functional UIs.
But, in the current backend design, both telemetry modules are not truly tied to user-driven event inputs (the backend uses deterministic sampling). Shock-test is the one that is more naturally parameter-driven because it consumes events + shock parameters.
